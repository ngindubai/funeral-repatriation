Repatriate Service — Template Diversification & Indexation Build Guide

A YMYL site about funeral repatriation cannot afford soft-deindexation. Right now every country hub renders from one Hugo template, every city page from one template, every guide from one template, every ashes/cremation/embassy page from one template. Same DOM order, same H2 sequence, same hero layout. Google will sample, cluster, and crawl-throttle. This guide instructs the build chat to introduce 5 distinct templates per content silo, 5 paired writing souls, deterministic rotation, and a phased back-fill across the live site.

Sensitivity rule above all. This is grief content. Diversification must never produce a "fun" or "marketing-flashy" treatment. Every variant remains compassionate, direct, professional. Variation lives in structure, not in tone.

Every country hub renders this exact sequence (verified from country-hub.html):

  2. Helpline strip
  3. Direct-answer block (where present)

  9. FAQ accordion
  10. Embassy contacts
  11. City list
  12. Final CTA

26 hubs, identical 12-section skeleton. Easy cluster.

2. Why one template causes deindexation (YMYL angle)

Standard near-duplicate detection plus a YMYL twist:

  • YMYL trust scoring. Funeral / death / repatriation triggers Google's YMYL bar. The algorithm wants visible E-E-A-T signals that aren't templated. Identical schema, identical FAQ block markup, identical author byline placement, page after page, suggests a content farm pattern. That depresses ranking even before deindexation.
  • Boilerplate ratio. The shared template chrome (helpline strip, repeated CTAs, repeated sidebar text) competes with the unique country-specific text. Pages with thin unique content get sampled out.
  • FAQ schema duplication. Currently FAQs are generated from the same 6 angles per country. Across 26 hubs that's 156 FAQs, many with near-identical question phrasing. Google merges these clusters and shows one country's snippet as canonical for several queries.

3. Design constraints

All variants must:

  • Use the existing palette (Playfair Display + Inter; navy / charcoal / gold accent already in main.css).
  • Maintain compassionate tone. No bright accents introduced solely for variety.
  • Keep the 24/7 helpline visible above the fold on every variant.
  • Honour all content hard rules: British English, no banned vocabulary (robust, seamless, comprehensive, crucial, meticulous, leverage, synergy, paramount, multifaceted, cornerstone, empower, catalyst, nestled, realm, tapestry, delve), no em dashes, no safety guarantees.
  • Preserve YMYL signals: source attribution, named author block, last-updated date, citation links to .gov / FCDO / embassy sources.
  • Render the same data fields from data/countries_repatriation.json — nothing dropped.

What changes per variant:

  • Section order.
  • Hero composition (image hero vs typographic hero vs answer-first hero vs map hero vs stat-strip hero).
  • Cost / timeline visualisation (table vs ladder vs split-card vs inline-text vs ledger-style).
  • FAQ placement (mid-page vs end vs interleaved).
  • Sidebar presence and content (sticky helpline rail vs document checklist rail vs none).
  • Schema.org type emphasis (Service vs HowTo vs FAQPage vs Article vs GovernmentService).

4. The 5 country-hub templates

Files in site/layouts/partials/countries/; the country-hub.html top-level becomes a slim router.

CH-A — The Family Brief

country-a-familybrief.html

Hero: Soft typographic hero, no large image. Country name, single sentence "We help families bring loved ones home from {country}." Helpline phone large beneath.

Sidebar: Sticky helpline + "What we'll do for you" 4-bullet card.

Section order: Hero → Direct answer (3 sentences max) → "What happens first" (immediate next steps) → Documents and processing → Cost (single-line range, breakdown collapsed by default) → Timeline as a brief paragraph not a 3-column grid → Local funeral director coordination notes → FAQ at end → Embassy contacts.

Visual signature: Quietest variant. Reads like a letter.

CH-B — The Step-by-Step

country-b-stepbystep.html

Hero: Small banner with country flag, status chip ("Repatriation: established route" / "Repatriation: complex"), single CTA.

Sidebar: Vertical step indicator, scroll-tracking through the 8 process steps.

Section order: Hero → 8 process steps as a single connected vertical journey (one card per step, no grid) → Cost rolled inline at the document step → Timeline shown as a single horizontal bar, not 3 cards → FAQ → Embassy contacts → Cities.

Visual signature: Linear journey. The 8 steps dominate the page.

CH-C — The Cost & Timeline Forward

country-c-costtimeline.html

Hero: Stat strip — typical days, typical cost, post-mortem likelihood, embassy support. No image.

Sidebar: Cost breakdown sticky table.

Section order: Hero stats → Detailed cost breakdown table (top of page, not buried) → Timeline ladder (best/typical/complex shown as a vertical ladder with day counts) → What drives cost up → What shortens timeline → Documents → Process summarised → FAQ at end.

Visual signature: Numerical, transparent. Pricing-first.

CH-D — The Q&A Hub

country-d-qahub.html

Hero: Clean white hero, H1 phrased as a question ("How do you bring a loved one home from {country}?"), 3-line direct answer beneath.

Sidebar: Floating mini-FAQ ToC.

Section order: Hero question + answer → "What do you do in the first 24 hours?" Q+A → "How long will it take?" Q+A with timeline embedded → "What will it cost?" Q+A with cost embedded → "What documents are needed?" Q+A → "What if there's a post-mortem?" Q+A → "Can we cremate locally instead?" Q+A → "Who pays?" Q+A → "What does the embassy actually do?" Q+A → "How do we start?" Q+A with helpline.

Visual signature: No "Process" / "Documents" / "Cost" H2s exist — every section is a question. Eliminates the boilerplate H2 fingerprint.

CH-E — The Authority Briefing

country-e-briefing.html

Hero: Long-form authoritative intro (200 words), no large image. Author byline (named senior repatriation specialist) + last-updated date front and centre.

Sidebar: Sources and references panel (FCDO link, embassy link, country gov funeral authority link).

Section order: Hero intro → Country overview (jurisdiction, legal framework, religious considerations) → Process narrative (prose, not numbered cards) → Documents listed inline within the narrative → Cost discussion in prose with a small inline table → Timeline discussion in prose → Edge cases (post-mortem, criminal investigation, repatriation of ashes only) → Local context (regional differences within the country) → FAQ at end → Citations list.

Visual signature: Highest-authority variant. Long-form, sourced. Best for YMYL E-E-A-T signals.

6. The 5 guide / silo-page templates

Apply the same 5 templates across guides/, bringing-ashes-home/, cremation-transfer/, embassy-contacts/. Files live in site/layouts/partials/silos/:

  • S-A — Process explainer. Linear narrative, sectioned by stage of grief / stage of process.
  • S-B — Comparison. Side-by-side: cremation vs repatriation; ashes-by-air vs ashes-by-courier; embassy vs consulate; what's covered vs what isn't.
  • S-C — Cost-led. Cost is hero. Everything else supports.
  • S-D — Q&A. Ten questions, ten answers, no other section structure.
  • S-E — Authoritative reference. Long-form, citation-rich, looks like a published reference document.
Embassy contacts deserve special attention. They are the most templated silo (essentially a directory). Use S-E heavily here — wrap each contact list inside an authority-rich narrative about how to actually use embassy services. The current "name + phone + address" grid is a deindexation magnet.

7. Content structure souls (5 per silo)

Souls live at workforce/content/souls/. The chat must read the relevant soul before drafting any prose. Each soul defines voice, opening pattern, paragraph rhythm, fact-claim style, and banned moves.

Country-hub souls

Soul 1 — The Family Companion

Default partner: CH-A

Voice: a senior repatriation coordinator writing directly to a grieving family. Short sentences. No technical jargon without immediate plain-English gloss. Opens by acknowledging the situation in one sentence, then moves to action.

Section openers (rotate): "The first thing to know is", "Most families in your situation", "Before anything else,", "When this happens,".

Banned: "we understand", "we're sorry for your loss" (cliché), "rest assured", "peace of mind", any marketing softener. Banned banned-vocabulary list applies (robust, seamless, etc).

Soul 2 — The Process Guide

Default partner: CH-B

Voice: methodical, step-by-step, calm. Each section is one beat in the timeline. Uses time markers ("Within 24 hours.", "By day 3.", "Around day 7.") as section openers. Never lists processes as bullet dumps; always as connected actions with their owner ("the local funeral director will obtain X").

Banned: parallel bullet structures repeated across sections — vary the phrasing within steps.

Soul 3 — The Practical Specifier

Default partner: CH-C

Voice: numbers-first. Every section opens with the relevant number ("Typical total cost from {country} is £4,800–£7,200."). Plain prose around it. Cost factors named explicitly. Does not bury the price.

Banned: vague ranges where a precise figure exists; "varies depending on circumstances" without specifying which circumstances.

Soul 4 — The Question Answerer

Default partner: CH-D

Voice: writes in answers. Every section literally a question. Uses second person directly. Answers in 2–4 sentences then a single supporting fact. No long expository paragraphs.

Pattern: question → 1-sentence direct answer → 2-sentence elaboration → 1 specific number or document name. Then move on.

Soul 5 — The Authority Specialist

Default partner: CH-E

Voice: senior practitioner addressing readers who want depth. Long-form (1,800–2,500 words). Cites sources inline ("the FCDO's death-abroad guidance, current edition"). Uses local authority names ("the Bangkok Metropolitan Administration's Office of Foreign Affairs handles…"). Names religious or cultural considerations where relevant.

Mandatory: at least 4 named external sources, byline, last-updated date, "Reviewed by" line where possible.

City-page souls

Use the same 5 souls scoped to city level. City pages must not repeat the country hub's facts; they must add genuine local detail (which hospital systems, which airports handle cargo, regional cost variance, local funeral director directory).

Silo souls (ashes / cremation / embassy / guides)

  1. The Calm Explainer — narrative, soft, S-A.
  2. The Comparator — every paragraph contrasts options, S-B.
  3. The Cost Specialist — money first, S-C.
  4. The Direct Answerer — questions only, S-D.
  5. The Reference Author — sourced, long-form, S-E.

9. Per-page workflow the chat must follow

  1. Read the front matter. Note template_variant and content_soul. If absent, compute from slug-hash modulo 5 and write back.
  2. Load the soul file at workforce/content/souls/country-soul-{N}.md (or city-soul / silo-soul). Read in full.
  3. Open one nearby existing page with the same template_variant. Compare opening sentences and FAQ phrasing. The new page must not echo prior structure word-for-word.
  4. Apply the existing content rules: British English, no banned vocabulary, no em dashes, no safety guarantees, source attribution on every factual claim, sensitive grieving-family tone.
  5. Write or rewrite the body following the soul's structure. Wordsmith first, then Chameleon humaniser pass.
  6. Verify the H2 set. No H2 may appear identically on more than 30% of pages within its variant bucket. Paraphrase boilerplate H2s.
  7. FAQ uniqueness. At least 3 of the 6 FAQs must be country-specific (mention country, region, or named local authority). The shared 6-angle FAQ template must be retired.
  8. Internal links. Vary the related-pages block visual style with the variant.
  9. Author + last-updated block. Required on every variant. Variant E requires a "Reviewed by" line.

12. QA, fingerprint test, indexation monitoring

Fingerprint test (script: scripts/fingerprint_audit.py)

  1. Crawl public/ after each build.
  2. For each page extract: H2 sequence, DOM-skeleton hash (class-name sequence), unique vs boilerplate text ratio, schema type used.
  3. Cluster by skeleton hash within each silo. Pass: no cluster > 25% of silo. Fail: any > 40%.
  4. Flag H2 strings appearing on >30% of pages within a variant bucket — paraphrase required.
  5. Flag pages with boilerplate ratio > 0.45 — body needs expansion.

YMYL signal audit

Beyond the structural fingerprint, weekly check that every YMYL page has:

  • Named author with credentials
  • Last-updated date in current quarter
  • At least 2 external authoritative sources (FCDO, embassy, WHO, country government)
  • Honest disclaimer where appropriate ("Requirements can change. Verify current rules with the embassy before proceeding.")

Indexation monitoring (GSC)

  • Track "Discovered, not indexed" + "Crawled, not indexed" weekly.
  • Sample 20 pages per variant via URL Inspection API monthly.
  • Compare indexation rate by variant; the variant outperforming on E-E-A-T pages keeps its share, weaker variants get reduced 5%.

13. Hard rules (do not violate)

Absolute don'ts

  • Never compromise sensitivity for visual variety. No bright marketing flair on grief content.
  • Never drop the 24/7 helpline from above the fold on any variant.
  • Never weaken YMYL signals (author, source, date) on any variant.
  • Never collapse two templates because they "feel similar". They are distinct on purpose.
  • Never let any variant's share fall below 15% or rise above 25%.
  • Never reuse the same FAQ block markup across all variants — match the variant.
  • Never skip the soul-load step. The chat must read the soul before drafting prose.
  • Never repeat banned vocabulary (robust, seamless, comprehensive, crucial, meticulous, leverage, synergy, paramount, multifaceted, cornerstone, empower, catalyst, nestled, realm, tapestry, delve).
  • Never use em dashes. Use commas, periods, colons.
  • Never make safety guarantees ("ensure", "guaranteed", "always safe"). Use "reduce risk", "in most cases", "typically".
  • Never back-fill more than 10 pages in a single chat session. Quality dies.

Absolute musts

  • Front matter on every country / city / silo page must include template_variant and content_soul.
  • Every page rewrite logs to data/backfill_register.json.
  • Variant E (Authority Briefing) is mandatory for the 12 LLM-audit priority pages.
  • Every variant CSS is scoped under .tpl-{silo}-{x}.
  • The fingerprint audit must run before every Surge deploy.
  • Schema.org primary type varies per variant (see §10).
  • Compassionate, professional tone is preserved across all variants.
  • British English throughout.
